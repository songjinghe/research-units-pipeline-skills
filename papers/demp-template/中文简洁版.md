# 中文简洁版正文

## 写法思路

这篇文章的写法主线很简单：

1. 先把问题说清楚，不讲复杂系统，只讲数据库实验为什么麻烦。
2. 再把现有工具分成几类，说明它们各自只解决了一部分问题。
3. 然后把我们的平台收束成一条很短的实验生命周期：`导入 -> 打包快照 -> 按需部署 -> 跑测试 -> 留历史记录 -> 做比较`。
4. 最后用三个场景把平台的价值讲明白，不去硬拼性能数字，而是强调“减少重复工作”和“更容易比较历史实验”。

所以这篇文章不是在讲一个新数据库，而是在讲一个更适合做数据库实验的外层平台。

## 标题

面向大规模数据库实验的版本化测试平台

## 摘要

数据库实验往往要反复经历导入大数据、部署数据库、运行测试和记录结果的过程。对于小数据集，这些步骤还算可控；但当数据规模达到数百 GB，或者测试需要反复切换代码和配置时，重复导入和重复部署的成本就会变得很高。现有工具通常只覆盖这一流程中的某一部分：有的擅长跑 workload，有的擅长数据版本管理，有的擅长真实 trace 回放，但很少有平台把导入、快照、部署、测试和比较统一起来。为此，我们实现了一个轻量级数据库测试平台，把导入后的数据打包成可复用快照，并提供 `import`、`archive`、`deploy`、`run` 和 `compare` 五个用户动作。平台会为每次实验保存快照、workload、配置和 Git commit 等历史信息，从而支持快速复用、按需恢复和结果比较。本文通过几个典型场景展示该平台如何减少重复实验开销，并让数据库测试更容易复现和回看。

## 1. 引言与动机

数据库已经很成熟，但数据库实验流程仍然很手工。一个典型实验通常从大规模导入开始，然后部署数据库实例，运行 workload，最后记录“这次到底用了哪个 Git commit、哪个配置、哪份数据”。如果数据很小，这套流程还可以接受；但如果数据很大，或者系统版本变化很频繁，重复导入和重复部署就会非常浪费时间。

现有工具各自解决了一部分问题。BenchBase、OLTPBench 和 HammerDB 主要负责执行 workload；DVC 和 lakeFS 擅长做数据快照和回滚；Oracle Real Application Testing 和 SQL Server Distributed Replay 适合真实 trace 回放。但它们并没有把导入、快照、部署、测试和比较连成一个统一流程。我们的目标就是把这些分散能力合并到一个实验平台里。

本文的核心不是提出一个新的数据库系统，而是提出一个更适合数据库研究和测试的实验平台。它的价值在于：让数据只导一次，后面的实验可以反复复用；让部署和测试分开，减少重复恢复的成本；让每次运行都能留下历史记录，便于后续比较。

## 2. 平台设计与原型状态

平台围绕四类持久对象组织：

- `snapshot archive`：保存导入后的数据快照
- `deployment record`：记录某个快照恢复出来的数据库实例
- `workload bundle`：保存一次实验要执行的测试脚本或 workload
- `run log`：保存运行结果以及对应的快照、配置和 Git commit 信息

平台提供五个面向用户的动作：

- `import`：把原始数据导入目标数据库
- `archive`：把导入后的状态冻结成快照
- `deploy`：按需把快照恢复成可测试环境
- `run`：执行 workload 并记录结果
- `compare`：在历史记录中比较两次实验

这个设计刻意做得很小，因为 demo 论文最重要的不是功能多，而是用户一眼就能理解平台在做什么。

从实现上看，我们的“版本感知”并不是很复杂的版本控制系统，而是更实用的历史记录管理：每次测试都记录 Git commit、配置和 workload，后面可以按这些标签回看和对比。这样做的好处是，用户不需要猜这次实验和上一次实验到底差在哪。

## 3. Demo 流程

### 场景 1：导入一次，多次测试

先把一份大数据导入一次，然后把导入结果打包成快照。后续如果要跑不同的 workload，直接从这个快照恢复即可，不需要每次都重新导入。这个场景强调的是：大数据导入是昂贵步骤，所以应该尽量一次完成、长期复用。

### 场景 2：只读测试和修改型测试

对于只读 workload，平台可以直接在已有部署上运行，不需要额外恢复。对于会修改数据库的 workload，平台在运行后可以恢复到干净快照，避免人工重建环境。这个场景强调的是：不同测试类型应该走不同恢复策略，不能所有实验都按同一套流程处理。

### 场景 3：历史记录比较

平台会把每次运行保存下来，用户可以比较不同 Git commit、不同配置、不同 workload 下的实验结果。这个场景强调的是：实验不只是“跑完一次”，而是要能回看、能对比、能追踪变化来源。

## 4. 相关工作与定位

Benchmark 工具、数据版本工具和 replay 工具都很成熟，但它们各有侧重：

- BenchBase、OLTPBench、HammerDB：擅长跑 workload
- DVC、lakeFS：擅长数据版本和回滚
- Oracle / SQL Server replay：擅长真实 trace 回放

我们的平台不是替代这些工具，而是把它们的优势收拢起来，放进一个面向大规模数据库实验的统一流程里。换句话说，我们做的是“实验生命周期管理”，不是“单点功能”。

## 5. 结论

我们提出了一个面向大规模数据库实验的版本化测试平台。它把导入、快照、部署、测试和比较连成一条流程，让大数据导入不必每次重来，让测试结果可以按历史回看，让重复实验更容易复现。

这篇论文最想表达的事情其实只有一句话：

> 数据库本身可能已经很成熟，但数据库实验流程仍然可以被做得更省事、更可追踪、更容易比较。
