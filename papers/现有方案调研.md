# 现有方案调研

## 1. 先说分析需求

如果把你们的工作定位成“数据库测试平台 demo 论文”，那分析需求不应该从“我们想了哪些创新点”出发，而应该从“一个真正可用的数据库实验平台到底要解决什么问题”出发。结合你们的 PET benchmark 场景，我认为至少有四类需求：

### 1.1 超大数据导入需求
- 数据规模可能达到几百 GB。
- 导入一次就很耗时，不能把每次测试都绑定到重新导数上。
- 需要把导入结果变成可复用的实验资产，而不是一次性中间产物。

### 1.2 导入、部署、测试解耦需求
- 有些测试是只读的，不应该每次都重做导入。
- 有些测试会改库，跑完以后需要快速恢复干净环境。
- 因此，平台不能只是“跑 benchmark”，而要能管理实验生命周期。

### 1.3 版本感知需求
- 既要看数据版本变化，也要看代码版本变化。
- 需要能够回答：哪个版本开始变慢了？哪个改动导致错误增多了？
- 这类能力比单次跑分更有论文价值，因为它直接服务于性能回归与正确性漂移分析。

### 1.4 场景化 workload 需求
- 你们的 PET benchmark 不是通用 OLTP，也不是通用 HTAP。
- 它带有明确场景：12306、放票/买票更新、路径/中转查询、时间窗约束。
- 因此平台需要支持自定义 workload 的组织、复现与对比，而不是只跑固定模板。

## 2. 现有方案调研

下面按“它们针对什么”“不足之处”“与你们的共性”“你们比它们好的地方”来整理。

## 2.1 BenchBase

### 针对什么
BenchBase 是一个多数据库 SQL benchmark 框架，核心流程是 `create / load / execute`，主要面向 JDBC 关系型数据库。它适合标准 OLTP / OLAP workload 的执行与对比。[BenchBase 项目页](https://db.cs.cmu.edu/projects/benchbase/)，[GitHub](https://github.com/cmu-db/benchbase)

### 不足之处
- 重点是 benchmark 执行，不是实验平台管理。
- 没有把导入结果做成显式可复用的部署快照。
- 没有把版本管理、回归分析、正确性漂移作为主线。
- 主要面向关系型数据库，不是为 PET 场景定制。

### 与你们的共性
- 都有阶段划分。
- 都把数据准备和测试执行分开。
- 都强调可复现的 workload 执行。

### 你们比它好的地方
- 你们把导入后的结果快照化、压缩化，可多次部署复用。
- 你们支持“只读测试可跳过部署、改写测试后可快速恢复”的实验流程。
- 你们围绕 PET benchmark 设计场景化 query，而不是通用 workload。
- 你们可以进一步做版本感知分析。

---

## 2.2 OLTPBench

### 针对什么
OLTPBench 是 BenchBase 的前身，面向 JDBC-enabled relational database 的多线程 benchmark / load generator，同样采用 `create / load / execute` 的流程。[GitHub](https://github.com/oltpbenchmark/oltpbench)

### 不足之处
- 仍然是传统 benchmark runner。
- 没有导入后快照复用的显式能力。
- 没有版本化实验仓库的设计。
- 不面向你们这种大规模场景数据与混合更新-查询 workload。

### 与你们的共性
- 三段式实验流程。
- 数据加载与测试分离。
- 面向数据库系统测试而不是纯应用测试。

### 你们比它好的地方
- 你们把 load 结果沉淀成可部署资产，而不是一次性中间态。
- 你们更强调大数据量下的重复实验成本控制。
- 你们能围绕 PET 场景做专门 query，而不是通用事务模板。

---

## 2.3 HammerDB

### 针对什么
HammerDB 是一个工业化数据库负载测试工具，支持 Oracle、SQL Server、Db2、MySQL、MariaDB、PostgreSQL，常用 workload 是 TPROC-C 和 TPROC-H。[官方文档](https://www.hammerdb.com/docs4.11/index.html)

### 不足之处
- 更像工业级负载测试器，不是实验平台。
- 关注 schema build、workload run、结果浏览，但不强调版本回归。
- 对大数据快照复用和场景化 workload 的支持不是主线。
- 主要是通用 TPC 风格基准，不是 PET 业务语义。

### 与你们的共性
- 都需要准备阶段和执行阶段。
- 都要处理 schema / 数据 / workload 的关系。
- 都服务于数据库性能评测。

### 你们比它好的地方
- 你们的平台更强调导入后的复用，而不是每次重来。
- 你们把部署和测试进一步解耦。
- 你们的 workload 是围绕 12306 的路径、余票、放票、买票和更新查询混合设计的。

---

## 2.4 db-benchmarks.com

### 针对什么
db-benchmarks.com 的开源测试框架更像一个工程化实验平台，特点包括自动数据加载、Docker 资源约束、自动清 cache、cold run / warm-up 控制、结果落盘和上传可视化。[framework](https://db-benchmarks.com/framework/)

### 不足之处
- 它更关注测试执行公平性和可复现性，不专门解决大规模导入后的快照复用。
- 没有把数据版本、代码版本、测试版本统一进一个版本感知分析框架。
- 目标更偏通用数据库 / 搜索引擎，不是 PET 业务场景。

### 与你们的共性
- 都是实验平台思路，不只是单个 benchmark。
- 都强调结果可复现、环境可控、测试流程工程化。
- 都支持多轮实验和结果管理。

### 你们比它好的地方
- 你们把“导入”做成可复用快照，特别适合几百 GB 数据。
- 你们不仅控制测试环境，还关心修改型测试后的快速恢复。
- 你们把 PET benchmark 场景嵌入平台主线，而不是只做泛化测试。

---

## 2.5 DVC

### 针对什么
DVC 是数据版本控制工具，重点是数据、代码、模型和实验结果的一体化版本管理、快照恢复与复现。[DVC 文档](https://doc.dvc.org/example-scenarios/versioning-data-and-models)

### 不足之处
- 它不是数据库 benchmark 平台。
- 不负责数据库部署、负载测试、查询延迟采集。
- 不直接解决冷启动、warm-up、query replay 等实验问题。

### 与你们的共性
- 都强调快照化。
- 都强调版本可恢复。
- 都强调实验复现。

### 你们比它好的地方
- 你们不是只版本化数据，而是把数据版本、平台版本和测试过程连起来。
- 你们的数据快照服务于数据库测试，而不是一般意义上的实验数据管理。
- 你们还要面向 PET workload 的性能评估和正确性分析。

---

## 2.6 lakeFS

### 针对什么
lakeFS 是更企业化的数据版本控制平台，强调数据分支、隔离测试、回滚和大规模数据版本控制。[官网](https://lakefs.io/)

### 不足之处
- 它是数据控制平面，不是数据库测试平台。
- 它不负责具体数据库 workload 的回放和性能采集。
- 它不处理 PET 场景的 query 设计。

### 与你们的共性
- 都需要数据版本管理。
- 都需要可回滚。
- 都强调大规模数据下的复用与隔离。

### 你们比它好的地方
- 你们把数据版本控制和数据库 benchmark 执行结合起来。
- 你们关注的是测试结果，而不仅是数据治理。
- 你们的版本切换直接服务于查询、更新和回归分析。

---

## 2.7 OLxPBench

### 针对什么
OLxPBench 主要评估 HTAP 场景，强调实时查询、fresh data、混合事务与分析工作负载。[Summary](https://www.benchcouncil.org/olxpbench/summary.html)

### 不足之处
- 主角仍然是 HTAP 引擎，而不是实验平台。
- 没有显式的数据快照复用机制。
- 没有你们这种 PET 场景化路径和票务查询语义。

### 与你们的共性
- 都强调更新与查询混合。
- 都关心 fresh data。
- 都关注真实世界的混合负载。

### 你们比它好的地方
- 你们更强调“导入后复用”和“改写后恢复”。
- 你们的场景更具体、更真实，不是泛化 HTAP。
- 你们能把路径/中转/票务变化做成专门 query。

---

## 2.8 HTAPBench

### 针对什么
HTAPBench 也是从 OLTPBench 衍生出来的，目标是评估能够同时承载 OLTP 事务和 OLAP 查询的引擎，并且不依赖 ETL。[GitHub](https://github.com/faclc4/HTAPBench)

### 不足之处
- 它是 HTAP benchmark，不是实验平台。
- 仍然偏向关系数据库及其混合负载。
- 没有把版本管理和性能回归分析作为主线。

### 与你们的共性
- 都是更新 + 查询混合负载。
- 都支持分阶段准备和测试。
- 都强调数据准备后再进行测试。

### 你们比它好的地方
- 你们的导入结果可以封装成可部署资产。
- 你们可以区分只读测试和改写测试，按需重部署。
- 你们可把 workload 绑定到 PET 场景，而不是 HTAP 泛化场景。

---

## 2.9 Oracle Real Application Testing

### 针对什么
Oracle 的 Real Application Testing 主要是 capture / replay 真实生产 workload，用来测试升级、补丁、硬件或调优对数据库行为的影响。[文档](https://docs.oracle.com/en-us/iaas/autonomous-database-serverless/doc/autonomous-real-application-testing.html)

### 不足之处
- 是厂商级功能，不是开放 benchmark 平台。
- 面向 Oracle 生态，通用性有限。
- 不提供 PET 场景化 query 设计。

### 与你们的共性
- 都强调真实 workload。
- 都强调回放和测试环境分离。
- 都希望评估系统变化带来的影响。

### 你们比它好的地方
- 你们不只是回放 workload，而是设计了 PET benchmark 场景。
- 你们把快照、部署、测试和版本管理统一起来。
- 你们可以做更细粒度的性能回归分析。

---

## 2.10 SQL Server Distributed Replay

### 针对什么
SQL Server Distributed Replay 用于捕获 trace，并在测试环境中分布式回放 workload，主要用来评估升级、硬件变化、操作系统变化和调优影响。[Microsoft Learn](https://learn.microsoft.com/en-us/sql/tools/distributed-replay/sql-server-distributed-replay?view=sql-server-ver17)

### 不足之处
- 是 SQL Server 专有生态。
- 依赖特定 trace 机制。
- 不处理 PET 场景 workload 设计。
- 不负责大数据导入复用快照。

### 与你们的共性
- 都是 capture/replay 思路。
- 都强调测试环境复现。
- 都关注性能和兼容性影响。

### 你们比它好的地方
- 你们的平台面向自定义 PET benchmark 数据集。
- 你们对导入和部署流程有更细的抽象。
- 你们还能把版本化分析纳入实验平台主线。

## 3. 我们的平台解决了什么痛点

把上面这些方案放在一起看，你们的平台最有说服力的点，不是“我们也做了测试”，而是：

### 3.1 大数据导入成本太高的问题
现有 benchmark 大多默认导入是一次性准备步骤，但在几百 GB 级数据下，反复导入会让实验成本爆炸。你们通过导入后快照化，把导入变成可复用资产，直接压低重复实验成本。

### 3.2 只读测试和改写测试混在一起的问题
很多现有工具并没有把“会改库的测试”和“不会改库的测试”区别对待。你们可以按测试类型决定是否重部署，既快又干净。

### 3.3 版本变化无法定位的问题
现有 benchmark 往往只给单次分数，难以解释为什么这个版本变慢、那个改动出错。你们的平台把数据版本、代码版本和测试结果联系起来，能做性能回归和正确性漂移分析。

### 3.4 场景化 workload 不够真实的问题（可快速接入自定义场景）
通用 benchmark 很多，但 12306 这种带路径、中转、余票变化、放票买票更新的场景很少被专门建模。你们的平台把 workload 扎到 PET 场景里，能更好地说明数据库系统在真实业务下的表现。

## 4. 为什么需要你们这样的测试平台

如果从论文角度总结，最好的论证方式是：

- 现有工具已经证明了 benchmark 的分阶段执行、数据版本管理和 workload 回放都各自重要。
- 但这些能力通常是分散的：有的管 benchmark，有的管数据版本，有的管 replay，有的管结果可视化。
- 你们的平台把这些能力统一起来，并且加上 PET benchmark 场景，使得大规模数据库实验不再需要每次从导入开始，也能在版本变化上做性能与正确性分析。

换句话说，**你们的价值不只是“跑测试”，而是“让大规模 PET benchmark 实验可持续、可复用、可定位问题”。**

## 5. Abstract 风格总结

我们提出一个面向 PET benchmark 的数据库测试平台，旨在支持超大规模数据集的高效导入、部署复用与版本感知分析。与传统 benchmark 框架主要关注一次性数据装载和 workload 执行不同，我们的平台将导入结果快照化，并将导入、部署与测试阶段解耦，以适应几百 GB 级别数据下重复实验成本过高的问题。平台进一步支持针对只读与改写测试的不同恢复策略，从而在保证实验干净性的同时降低测试开销。此外，平台将数据版本、代码版本和测试结果统一关联，使得研究者能够追踪不同版本对查询性能、更新开销以及结果正确性的影响，进而进行性能回归与正确性漂移分析。围绕 12306 场景设计的 PET benchmark workload 进一步体现了该平台的实用性，使其能够服务于真实业务驱动的数据库评测、实验复现与系统迭代。该平台为大规模、场景化、版本化的数据库测试提供了一条可复用的工程路径。
